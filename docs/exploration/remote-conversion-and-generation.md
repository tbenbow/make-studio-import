# Remote Conversion & Generation — Exploration Brief

## Context

The `make-studio-importer` project has two proven workflows that currently require Claude Code running locally in a terminal:

1. **Site Generation** — A vibe prompt produces 4 self-contained HTML variations (Tailwind CDN + Google Fonts, no frameworks). The user picks one, and it gets converted into Make Studio blocks.

2. **HTML → Blocks Conversion** — A static HTML page is decomposed into semantic sections (Hero, Features, CTA, etc.), each converted to a Handlebars template with JSON field definitions, a theme.json with design tokens, and content populated via the Make Studio API.

Both workflows produce fully editable, themeable sites in Make Studio. Blocks are portable between sites because they use semantic design tokens rather than hardcoded values.

## Goal

Explore making these workflows available as product features — no terminal required. Two entry points:

### Entry Point 1: Upload Flow

A user uploads an `index.html` file (generated by Claude, hand-coded, or exported from any tool) and gets a Make Studio site back.

**What the conversion actually does:**

1. Reads HTML, identifies `<section>` boundaries
2. Extracts colors → maps to 10 system color tokens (brand, base, fg, border, etc.)
3. Extracts typography → maps to heading/body tier system (xl through xs)
4. For each section:
   - Creates a Handlebars template with Tailwind classes using semantic tokens
   - Builds a JSON field definition (text, textarea, wysiwyg, image, items, group, select, link, etc.)
   - Sets default values for every field
   - Replaces buttons with a `{{> Button}}` partial
5. Builds `theme.json` (fonts, systemColors, customColors, palette, typography, prose, buttons)
6. Syncs everything via the Make Studio API (createBlock, updateSiteTheme, createPage, setPageContent)

**Key insight:** Steps 1-5 are pure text-in/text-out. Only step 6 requires API access. This means the LLM work and the API work are cleanly separable.

### Entry Point 2: Remote Site Generation

Replicate the local generation experience: user provides a vibe prompt, gets 4 HTML previews, picks one, conversion runs automatically.

**What this adds beyond the upload flow:**

- HTML generation from a prompt (stateless, no tools needed)
- Preview rendering (could be server-side screenshots or iframe previews)
- A selection UI
- Then the same conversion pipeline

## Architecture Considerations

### What the LLM needs as context

The conversion quality depends on ~30-50k tokens of guide material:

- `docs/capabilities/conversion/guide.md` — The conversion workflow
- `docs/capabilities/design/guide.md` — Color token mapping, typography tiers, button variants
- `docs/capabilities/blocks/guide.md` — Block creation rules
- `docs/capabilities/blocks/checklist.md` — Field type rules, defaults, gotchas
- `docs/references/*.md` — Gold-standard block examples (Hero, Features, Stats, Navbar, Footer, CTA, Testimonial, Button)
- Field type reference: text, textarea, wysiwyg, image, video, link, select, number, date, toggle, items, group

This context is currently read from disk by Claude Code. In a service, it would be bundled into the system prompt.

### API surface needed

The backend service needs to call these Make Studio API endpoints:

| Operation | Endpoint |
|-----------|----------|
| Create site | `POST /sites` |
| Update theme | `PATCH /sites/:id` (theme field) |
| Create block | `POST /blocks` |
| Create partial | `POST /partials` |
| Create page | `POST /pages` |
| Set page content | `PUT /pages/:id/content` |
| Create layout | `POST /layouts` |
| Deploy preview | `POST /sites/:id/deploy` |

All use bearer token auth. The token is per-site.

### Execution options

| Option | Pros | Cons |
|--------|------|------|
| **Claude API direct** | Simplest. Single API call with structured output. No tool use needed for conversion. | One-shot — no iteration loop. Large context per request. |
| **Claude Agent SDK** | Can give the agent Make Studio API tools. Supports multi-turn iteration. | More complex to set up. Needs tool definitions. |
| **Self-hosted (closet server)** | Full control, no per-request cost after hardware. Can run continuously. | Hardware management, model hosting complexity. |
| **Hybrid** | Claude API for the LLM work, your server for API calls and orchestration. | Two systems to maintain. |

### The iteration question

Locally, conversion is interactive — Claude makes judgment calls, the user reviews, adjustments happen in conversation. In a service:

- **One-shot mode**: Upload HTML → get site. Fastest, simplest. Quality depends entirely on the guides being comprehensive enough.
- **Preview-and-adjust mode**: Upload → preview → request changes → re-convert. Requires maintaining a conversation.
- **Full conversation mode**: Like Claude Code but in a web UI. Most powerful, most complex.

Recommendation: start with one-shot to validate quality, add preview step if needed.

## What to validate first

1. **Can the conversion run in a single Claude API call?** Bundle the guides as system prompt, send the HTML as the user message, request structured JSON output (theme + blocks + content). If the quality matches what Claude Code produces interactively, the upload flow is viable.

2. **What's the right output format?** The service needs to return something the Make Studio backend can consume. Options: (a) the service calls the API directly, (b) the service returns a structured payload that Make Studio's backend processes.

3. **Where does this live?** As a Make Studio API endpoint (`POST /sites/from-html`), as a separate microservice, or as a feature in the Make Studio admin UI?

## Phase 1 Findings: HTML Generation via Claude API

Tested generating Tailwind HTML through the Claude API with a design-focused system prompt adapted from Claude Code's frontend-design skill.

### The frontend-design skill is just a prompt

The skill is ~40 lines of markdown that pushes for bold aesthetic choices and bans generic AI aesthetics (no Inter, no purple gradients, no predictable layouts). It's fully portable to an API system prompt. See `scripts/test-generation.ts` for the adapted version.

### Prompt specificity drives quality, not the model

We tested with both Sonnet and Opus. The model difference was marginal. What made the dramatic quality difference was **how specific the aesthetic direction was in the prompt**. A vague "make it bold" produced generic results. A detailed vibe spec (exact fonts, color hex codes, named CSS effects, section concepts, content ideas) produced excellent results comparable to local Claude Code generation.

### The vibe brief is the key artifact

The most important input to generation is a detailed aesthetic brief that includes:
- Specific font choices (e.g., "Orbitron for headings, Bebas Neue for subheadings, Share Tech Mono for body")
- Exact color palette with hex values
- Named CSS effects (scanlines, glitch, grain, etc.)
- Section concepts with specific content ideas
- Tone and personality direction

Without this, even Opus produces safe, generic output. With it, Sonnet produces great results.

### Recommended approach: two-stage generation

1. **Design brief generation** — Claude takes wizard intake (business name, what they do, audience, CTA, differentiator) and produces a detailed aesthetic brief. This is a short, fast API call.
2. **HTML generation** — Claude takes the design brief and generates the full-page HTML. This is a longer call but has very specific direction.

**Fallback option:** Pre-write a library of design briefs for common business types. Claude picks the best match and customizes it. These are cheap to produce and guarantee quality.

### Test script

`scripts/test-generation.ts` — Takes wizard-like intake params, calls Claude API, outputs HTML to `output/<slug>/index.html`. Supports `--vibe` param for aesthetic direction.

## Revised Architecture: Skip the Importer

Make Studio already has `POST /blocks/convert-code` which takes Tailwind HTML and returns Handlebars templates + field definitions. The full pipeline can run inside Make Studio:

```
Wizard intake
  → Stage 1: Claude generates design brief (fast, ~500 tokens out)
  → Stage 2: Claude generates full-page Tailwind HTML (using brief)
  → Server: parse HTML for fonts + colors → build theme
  → Server: split by <section>/<nav>/<footer>
  → For each section: /blocks/convert-code → Handlebars + fields
  → Assemble page (same as current page generation compiler)
  → Deploy preview
```

For generated HTML, theme extraction doesn't need Playwright/DOM inspection — fonts and colors are right in the source (Google Fonts links + Tailwind config block). Deterministic parsing may suffice.

### Key difference from current page generation

Current flow: select from existing block library → populate content.
New flow: generate bespoke HTML per section → convert to blocks → populate.

Every site gets unique blocks tailored to the content, not constrained by a pre-built library.

## Related: Block Library & Onboarding

The existing onboarding system based on block composition with recipes remains valuable as a fast path. Direct generation is a premium path for higher quality results. They can coexist — recipes for speed, generation for quality.
